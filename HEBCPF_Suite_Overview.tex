\section{Release Scope}

This release packages the v4 HEBCPF solver in two forms:

\begin{center}
\begin{tabular}{@{}p{0.28\textwidth}p{0.28\textwidth}p{0.34\textwidth}@{}}
\toprule
\textbf{Folder} & \textbf{Solver} & \textbf{Use it when}\\
\midrule
\textcolor{mexcol}{\code{HEBCPF_MEX_v4_20260715}} & Windows x64 MEX-accelerated v4 &
You want the fastest shipped build on a supported Windows MATLAB installation.\\
\textcolor{matcol}{\code{HEBCPF_matlab_v4_20260715}} & Pure MATLAB v4 &
You need portability, easier inspection, or a no-compiler/no-MEX workflow.\\
\bottomrule
\end{tabular}
\end{center}

Both solvers use the same HEBC formulation, MATPOWER-format cases, v4 numerical
parameters, and user-facing drivers. They should not be placed on the MATLAB path at the
same time because they contain functions with overlapping names.

\subsection{Relationship to earlier releases}
Release 2026.07.15 is a benchmarked maintenance release of the v4 line, not a new
mathematical method. Relative to the 2026.07.14 v4 release, it retains the formulation,
numerical defaults, compiled hot paths, and queue scheduler; the substantive driver change
is serializable checkpoint/resume support for long \code{parfor} and queue \code{parfeval}
runs, plus the retained MEX barrier driver. Earlier v2/v3 packages and the 2026.07.14 v4
package remain useful for reproducing historical results and are available through the
corresponding GitHub tags. Do not mix solver folders from different releases on one MATLAB
path.

\section{Core v4 Features}

\begin{itemize}[leftmargin=1.4em]
  \item \textbf{Consensus Pad\'e-pole selection.} The holomorphic-to-continuation hand-off
        uses a pole estimate agreed across several buses, reducing spurious single-bus pole
        choices.
  \item \textbf{Deterministic keyed deduplication.} Candidate solutions are matched with a
        fixed random signature and an L1 tolerance, avoiding the old linear scan over the
        full solution matrix.
  \item \textbf{\code{slope_max = 4e5}.} The v4 folders ship with the requested turning
        point slope threshold in \code{solver_params.m} and in the parallel runbooks.
  \item \textbf{Global \code{parfeval} work queue.} The recommended driver keeps one
        in-flight trace per equation and submits new work as soon as a worker finishes.
        This preserves \code{VBook} pruning while avoiding row-barrier idle time.
  \item \textbf{Checkpoint/resume support.} Long parallel searches periodically save
        \code{temp_result.mat}; resuming rebuilds futures, pools, and deduplication state.
  \item \textbf{Cached/vectorized hot path.} V4 reduces repeated sparse analysis,
        holomorphic assembly work, and Pad\'e allocation overhead.
\end{itemize}

\section{Benchmark Comparison}\label{sec:benchmark}

The released benchmark compares the two shipped v4 packages, not earlier solver releases.
It ran the queue \code{parfeval} driver at nominal load on all 20 bundled cases through
57 buses using MATLAB R2022a on \code{PCWIN64} with 23 local workers. Parallel-pool startup
(44.46~s) was performed once before the timed loop and is excluded; the reported per-case
wall times include preprocessing and tracing. The benchmark driver is
\code{run_benchmark_up_to_57bus_20260715.m}; the committed machine-readable results are in
\code{benchmark_results_20260715/20260715_194046/benchmark_summary_all.csv}.

\begin{center}\small
\begin{tabular}{@{}lrrrr@{}}
\toprule
\textbf{Solver} & \textbf{Cases} & \textbf{Total wall (s)} & \textbf{Trace wall (s)} & \textbf{Solutions}\\
\midrule
MEX v4 & 20 & 718.656 & 716.062 & 3256\\
Pure MATLAB v4 & 20 & 1272.966 & 1271.222 & 3256\\
\bottomrule
\end{tabular}
\end{center}

Across the complete suite, the MEX package took $1.77\times$ less wall-clock time
(43.5\% less elapsed time) and was faster on 14 of 20 cases. This is a package-level
comparison: MEX v4 uses the compiled Pad\'e/holomorphic kernels and its shipped KLU path,
whereas pure MATLAB v4 uses only \code{.m} code and its dense corrector. Very small cases
are overhead-sensitive, so pure MATLAB was faster on six of them; do not treat the aggregate
ratio as a platform-independent or serial-mode speedup.

Both solvers returned the same solution count for every case. A one-to-one, order-independent
comparison of the saved solution sets gave a maximum nearest-solution distance of
$3.154\times10^{-7}$, below the shared keyed-deduplication tolerance of \code{4e-7}.

\section{Choosing A Mode}

Each solver has three main collection modes:

\begin{description}[leftmargin=0em,style=nextline]
\item[\code{serial}] Uses \code{runVBook_hybrid_2023.m}. It is the easiest mode to debug
and needs no Parallel Computing Toolbox.
\item[\code{parfor}] Uses \code{runVBook_hybrid_parallel.m}. It traces one row of
\code{VBook} at a time and waits for all equations in that row to finish.
\item[\code{parfeval}] Uses \code{runVBook_hybrid_parfeval.m}. It is asynchronous and keeps
workers busy without waiting for the slowest equation in each row. This is the recommended
mode for \code{case30}, \code{case39}, \code{case57mod}, \code{case57}, and larger systems.
\end{description}

The MEX folder also keeps \code{runVBook_hybrid_parfeval_barrier.m}. That file is a
retained comparison driver, not the recommended default.

\section{Resuming A Ceased Search}

The parallel v4 runbooks write periodic checkpoints to \code{temp_result.mat}. To resume:

\begin{enumerate}[leftmargin=1.6em]
  \item Return to the same solver folder.
  \item Use the same case, load factor, and driver mode.
  \item Load \code{temp_result.mat}.
  \item Run the matching runbook.
\end{enumerate}

\begin{lstlisting}[style=mat]
cd('<path>/HEBCPF_MEX_v4_20260715')       % or the pure MATLAB folder
addpath(pwd)
load temp_result.mat
wait = 0;
run('runVBook_hybrid_parfeval.m')         % queue parfeval checkpoint
% run('runVBook_hybrid_parallel.m')       % parfor checkpoint
\end{lstlisting}

For the MEX retained barrier driver:

\begin{lstlisting}[style=mat]
load temp_result.mat
wait = 0;
run('runVBook_hybrid_parfeval_barrier.m')
\end{lstlisting}

For serial runs, save the workspace manually before stopping:

\begin{lstlisting}[style=mat]
save('my_checkpoint.mat', '-v7.3')
load my_checkpoint.mat
wait = 0;
run('runVBook_hybrid_2023.m')
\end{lstlisting}

Internally, the resume path checks \code{VBook} and \code{Zsave}. If \code{Zsave} contains
extra preallocated columns, it is trimmed to the book-kept solution count. The queue
\code{parfeval} driver rebuilds pending work from \code{VBook == 0}; runtime futures and
parallel-pool objects are not restored from the checkpoint.
